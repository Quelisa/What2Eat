# Design: 修复卡片阴影底部"灰条"

## 问题诊断（视觉机制）

```
                  挂件卡片（圆角 22px）
                ╭─────────────────╮
                │                 │
                │   ...内容...    │
                │                 │
                ╰─────────────────╯  ← 圆角底
                  ░░░░░░░░░░░░░     ← y-offset 12px 把阴影压到下方
                    ░░░░░░░░       ← 模糊 40px 的阴影
                      ░░░         ← 圆角处被"啃掉"
                                  ← 中段最浓 = 用户看到的"一小段冷灰"
```

`box-shadow` 投射的是**卡片矩形 outset 后的模糊**，圆角让阴影在两侧迅速衰减，正下方却由于 `y-offset` 被推出最远——视觉效果是底部正中一道窄而浓的暗段，恰好落在 `reportSize()` 上报的窗口高度（`rect.height + 24`）的底部 12px 透明区里，肉眼读作"挂件下方有一段灰条"。

## 方案对比

| 方案 | 改法 | 立体感 | 圆角啃边 | 渲染成本 | 风险 |
|---|---|---|---|---|---|
| A. 去掉 box-shadow | `box-shadow: none` | ❌ 完全消失 | — | 最低 | 卡片"贴在桌面"质感丢失 |
| B. 降不透明度 | `0 12px 40px rgba(0,0,0,0.08)` | 弱 | 仍有 | 同 | 改善但症状仍可能可辨 |
| C. 缩小对称阴影 | `0 4px 16px rgba(0,0,0,0.10)` | 中 | 弱 | 低 | 立体感保留，啃边大幅减弱 |
| D. 环境光阴影 | `0 0 20px rgba(0,0,0,0.10)` | 中（无方向） | 几乎无 | 低 | 失去"光从上方来"的方向感 |
| **E. drop-shadow** | `filter: drop-shadow(0 8px 16px rgba(0,0,0,0.12))` 替代 box-shadow | 中–强 | **无**（沿形状投影） | 略高 | 完美解决；硬件加速影响极小 |

## 采纳：E + 微调（drop-shadow）

**核心理由**：`box-shadow` 是按元素矩形外扩模糊；`filter: drop-shadow` 是按**渲染后的形状**（含圆角，含子元素）投影。后者从机制上消灭"圆角啃边"问题——阴影沿着卡片的圆角弯下去，而不是从矩形边缘起步再被圆角剪。

具体值（实施时按视觉验证微调，最终采纳）：

```css
/* widget-compact.html .widget */
filter: drop-shadow(0 2px 10px rgba(0, 0, 0, 0.08));
/* 移除原 box-shadow */

/* wallpaper.html .glass-card */
filter: drop-shadow(0 3px 14px rgba(0, 0, 0, 0.10));
/* 移除原 box-shadow */
```

> 注：design.md 初稿提议 `0 6px 18px / 0 8px 22px`，实施时视觉确认仍能看到偏下的深阴影，遂将 y-offset 砍半、模糊和不透明度同步下调。最终值仍在 spec 约束内（y ≤ 10、blur ≤ 24、opacity ≤ 0.18）。

参数取舍：
- **y-offset 减小**（12 → 6 / 16 → 8）：阴影离卡片更近，避免重新出现"被窗口底部切掉"的问题
- **模糊半径减小**（40 → 18 / 56 → 22）：与新 y-offset 匹配，整体阴影更紧凑
- **不透明度略降**：drop-shadow 沿形状投影，每像素都生效，需要稍降避免视觉过重

## 副作用排查

### 1. `filter` 会让子元素脱离合成层吗？

会。卡片树会被合成成一张位图后再 drop-shadow。但卡片内只有静态文字和按钮，没有 GPU 动画或 transform-3d，影响可忽略。`hover` 上的 transform 是按钮自己的，不受卡片 filter 影响。

### 2. `filter` 会影响 `.drag-handle` 的 `position: absolute` 吗？

不会。drop-shadow 是合成阶段的视觉滤镜，不改变布局；拖拽条仍在卡片顶部 22px 处，事件位置正确。

### 3. `reportSize` 上报的尺寸还准吗？

`getBoundingClientRect()` 不计 box-shadow 也不计 filter——但**因为 drop-shadow 投影更短** (6+18=24px ≪ 12+40=52px)，原来阴影被切的"灰段"问题就不存在了。即使 24px 阴影被窗口边缘切到一点，肉眼也基本看不出。

如果将来想完全消除可能的边缘切痕，可在 `reportSize` 里给上报高度多加 `+filterPad`（例如 `+30`）作为缓冲；本 change 不做，留待观察。

### 4. 如果未来有人按 `box-shadow` 模式继续往卡片加 hover 阴影怎么办？

通过新增 Requirement「卡片阴影需柔和、不在挂件底部留下可辨硬色段」把视觉契约固化，下一个 change 再加 hover 阴影时也得过同样的视觉验收。

## 验证场景

| 场景 | 期望 |
|---|---|
| 默认紧凑版，挂件停右下角 | 卡片下方无可辨识的灰色段；柔和环绕阴影 |
| ⌥⌘T 切换到完整版 | 同样无灰条 |
| 多次 ⌥⌘R 刷新 | 阴影渲染稳定 |
| 拖到屏幕中央（远离边角） | 阴影沿圆角对称分布 |
| 切到激活模式 (⌥⌘E) | 阴影外观与贴桌面模式一致（仅窗口层级不同） |

## 风险

- **低**：filter 在 hi-DPI 屏可能与 box-shadow 渲染细节有像素级差异——肉眼无感。
- 若用户后续上报"换了显示器后阴影看起来又有问题"，第一步排查就是 `reportSize` 的 padding 是否够；本 change 在 design.md 留下追加 buffer 的预案。
