# desktop-widget — 桌面挂件能力规约

记录"What2Eat 桌面挂件"作为一项能力当前应该满足的所有需求。每条需求带可验证的场景。

## 需求 R1：菜品推荐

**用户能在挂件上一键获得一份午餐 / 晚餐推荐。**

### 场景 R1.1：默认推荐（启动即出）

- **当**用户启动挂件时
- **那么**挂件应在 1 秒内自动出现一条菜品推荐 + 一段对应文案
- **且**当前时段（中餐 / 晚餐）应根据系统时间自动判定（10:00–14:59 中餐，其他晚餐）

### 场景 R1.2：换一家

- **当**用户点击挂件上的「吃啥」/「换一家」按钮时
- **那么**挂件应立即给出一道新的菜品 + 一条该菜品对应的文案
- **且**新结果与上一次结果**可以**相同（避免菜品池小时无法刷新）
- **且**文案应从该菜品的多条文案中随机取一条

### 场景 R1.3：手动切换星期 / 餐次

- **当**用户在挂件上把星期切到 X、餐次切到 Y 后点「换一家」
- **那么**推荐应仅从「在 X 星期 Y 餐次可吃」的菜品集合中产生

## 需求 R2：菜品池与时段规则

**菜品池硬编码在 HTML 中，遵守显式的时段限制规则。**

### 场景 R2.1：基础菜品池

- 所有 HTML 中的 `dishes` 数组**必须**包含至少这 11 个菜品：烧鸭、瓦罐汤、喜家德、老乡鸡、麦当劳、自选菜、酱骨、跷脚牛肉、寿喜锅、云南生烫米线、轻食
- 每个菜品**必须**至少配 2 条文案

### 场景 R2.2：麦当劳限定

- **当**用户处于「非周五」**或**处于「中餐」时段
- **那么**麦当劳**不应**出现在候选集中

### 场景 R2.3：老乡鸡限定

- **当**用户处于中餐时段
- **那么**老乡鸡**不应**出现在候选集中

### 场景 R2.4：候选为空兜底

- **当**当前 (星期, 餐次) 组合下没有任何可推荐菜品
- **那么**挂件应显示提示文本（例如"今天换个口味吧"），**不得**抛出异常或停在空白

## 需求 R3：双层级模式

**挂件提供"贴桌面"和"激活"两种状态，由用户用快捷键显式切换。**

### 场景 R3.1：默认贴桌面

- **当**Hammerspoon 启动加载 `init.lua` 时
- **那么**挂件**必须**进入"贴桌面"模式（窗口层级 = `desktop + 1`），任何应用窗口都能盖住它
- **且**鼠标点击会穿透到桌面（这是 macOS `kCGDesktopWindowLevel` 的固有行为，文档中需明示）

### 场景 R3.2：激活模式可点击

- **当**用户按 `⌥⌘E`
- **那么**挂件层级应升到 `floating`，浮在普通应用之上
- **且**所有按钮（换一家、切换星期 / 餐次、拖拽条）**必须**可响应鼠标事件

### 场景 R3.3：失焦自动收起

- **当**挂件处于激活模式且失去主窗口焦点
- **那么**挂件**应**自动回到贴桌面模式

## 需求 R4：拖拽与定位

**用户能用鼠标把挂件拖到任意位置，松手时吸到最近的角落和屏幕。**

### 场景 R4.1：仅拖拽条响应

- **当**用户在挂件**最顶部 22–36px 的拖拽条**按下鼠标左键并拖动
- **那么**挂件**必须**跟随鼠标移动
- **且**按在拖拽条**之外**的区域（按钮、菜品文字、控件）按下不应触发拖拽

### 场景 R4.2：自动吸附

- **当**用户拖动挂件后松开鼠标
- **那么**挂件**必须**根据中心点所在屏幕和象限，吸到该屏幕的「左上 / 右上 / 左下 / 右下」之一
- **且**新位置**必须**写入 `~/.hammerspoon/widget-state.lua` 持久化

### 场景 R4.3：键盘定位

- **当**用户按 `⌥⌘1` / `⌥⌘2` / `⌥⌘3` / `⌥⌘4`
- **那么**挂件**应**贴到当前屏幕的左上 / 右上 / 左下 / 右下角
- **当**用户按 `⌥⌘ →` / `⌥⌘ ←`
- **那么**挂件**应**循环切换到下一块 / 上一块屏幕，并保持当前角落

## 需求 R5：自适应尺寸

**挂件高度跟随内容高度变化，不出现内部滚动条。**

### 场景 R5.1：内容驱动高度

- **当**网页内容（推荐卡片显隐、文案长短）变化时
- **那么**网页**必须**通过 `webkit.messageHandlers.hsbridge.postMessage({type:"resize",height,width})` 报告新尺寸
- **且**Hammerspoon 收到 `resize` 消息后**应**调整窗口尺寸（高度限制 120–1200px、宽度限制 200–1200px），并按当前 corner 重新定位

### 场景 R5.2：永不出现滚动

- HTML 的 `body`/`html` **必须**设 `overflow: hidden`
- **当**内容超出当前窗口高度时，**应**通过 R5.1 的机制扩大窗口而**不是**让用户滚动

## 需求 R6：双设计可切换

**用户可在「完整版」和「紧凑挂件版」之间切换，记忆选择。**

### 场景 R6.1：默认紧凑

- **当**用户首次启动 Hammerspoon（无 `widget-state.lua` 或其中无 `design`）时
- **那么**挂件**必须**加载 `widget-compact.html`（360×200）

### 场景 R6.2：一键切换

- **当**用户按 `⌥⌘T`
- **那么**挂件**必须**切换到另一份 HTML 设计（`compact` ↔ `full`）
- **且**新选择**必须**持久化
- **且**窗口尺寸**应**重置为该设计的默认值，等待 R5.1 自适应再调整

## 需求 R7：状态持久化

**屏幕、角落、设计三项状态在 Hammerspoon 重启后保持。**

### 场景 R7.1：状态文件

- 状态**必须**写入 `~/.hammerspoon/widget-state.lua`，格式为可被 `dofile` 加载的 Lua 表
- 字段**必须**包含：`screenIndex`（整数）、`corner`（"top-left" / "top-right" / "bottom-left" / "bottom-right" 之一）、`design`（"compact" / "full" 之一）

### 场景 R7.2：缺字段兼容

- **当**状态文件存在但缺少某字段时
- **那么****应**用默认值（`screenIndex=1`、`corner="bottom-right"`、`design="compact"`）填充，不报错

## 需求 R8：快捷键约定

**所有全局快捷键统一使用 `⌥⌘` 修饰键，避免与 macOS 系统手势冲突。**

| 快捷键 | 功能 |
|---|---|
| `⌥⌘ E` | 激活 / 贴桌面 切换 |
| `⌥⌘ T` | 切换设计（完整 / 紧凑） |
| `⌥⌘ R` | 刷新当前 HTML |
| `⌥⌘ W` | 重载 Hammerspoon 配置 |
| `⌥⌘ ← / →` | 切换屏幕 |
| `⌥⌘ 1 / 2 / 3 / 4` | 贴左上 / 右上 / 左下 / 右下角 |

新增快捷键**必须**沿用 `⌥⌘` 修饰键，并补充进此表。
